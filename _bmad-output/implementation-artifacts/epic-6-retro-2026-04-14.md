## 3. Análise Profunda por Story

### Story 6.1: Cadastro e Atribuição de Plano por Tenant

**Padrões Consolidados:**
1. Separação clara domain/application/infrastructure em novo módulo `plans`
2. Entidades minimalistas: catalogo de planos, atribuição vigente, histórico append-only
3. Transição atômica: encerra `effective_to` do plano anterior E cria novo com `effective_from` na mesma transação
4. Auditoria em nível de evento: cada mudança registra `actor`, `correlation_id`, `tenant_id`, timestamp

**Breakthroughs:**
- Schema Drizzle com constraint `UNIQUE (tenant_id, effective_to is null)` garante máximo um plano ativo por tenant
- Use case `get-active-tenant-plan` reutilizável por Stories 6.2/6.3 sem reconsultar BD

**Dívida Técnica Identificada:**
- Nenhuma crítica; policy docs de operação (como lidar com downgrade de plano) ainda pendente

### Story 6.2: Enforcement de Capacidades por Plano

**Padrões Consolidados:**
1. Catálogo de capacidades em código (`capabilities.ts`): `PLAN_CAPABILITIES` é source of truth (não em BD)
2. Enum tipado `Capability` com 6 capacidades iniciais: BATCH_INGESTION, EXTERNAL_INTEGRATIONS, PDF_MULTIPAGE_PROCESSING, ADVANCED_AUDIT, PORTAL_EMPLOYEE_ACCESS, COMMERCIAL_GOVERNANCE
3. Guard `enforceCapability(tenantId, capability)` que lança `CapabilityForbiddenError` quando bloqueado
4. Integração em handler: adicionar chamada `enforceCapability()` ANTES da lógica de domínio, SEM refatorar estrutura

**Breakthroughs:**
- Endpoint `GET /api/v1/tenants/me/capabilities` retorna lista de capabilities habilitadas para frontend feature flags
- HTTP 403 estruturado com `code: "CAPABILITY_FORBIDDEN"`, `upgrade_hint` sugestivo
- Auditoria de bloqueio registrada como evento `plans.capability.blocked.v1`

**Review Findings:**
- [RESOLVED] Auditoria integrada no `enforceCapability` com `actor` e `correlationId` via injeção
- [RESOLVED] Tipagem de `PLAN_CAPABILITIES` com Readonly e type para `PlanCode`
- [RESOLVED] Fallback text genérico para `upgrade_hint` quando capability não tem dica customizada
- [RESOLVED] Otimização `Promise.all` para `resolveTenantRole` + `getActiveTenantPlan`

### Story 6.3: Telemetria de Uso por Capacidade

**Padrões Consolidados:**
1. Schema `tenant_capability_usage` com `id`, `tenant_id`, `capability`, `usage_count`, `period_date`
2. Exportação em `src/lib/db/schema/index.ts` + migração Drizzle automática
3. Serviço `logCapabilityUsage` chamado de `enforceCapability` (assíncrono, best-effort)
4. Endpoint `GET /api/v1/platform/telemetry/capabilities` filtrável por tenant/capability/period

**Breakthroughs:**
- Logging assíncrono: falha em registrar telemetria NÃO bloqueia a requisição de usuário
- Integração com enforcement: cada sucesso de capability gera evento `plans.capability.used.v1`
- Importação de ApiMeta estendida com campo `plan_code?` opcional para respostas comerciais

**Debug Log:**
- Corrigido "circular reference" em Vitest movendo `vi.mock` para `vi.hoisted`
- Corrigido envelope de API estendendo `ApiMeta` para suportar `plan_code`
